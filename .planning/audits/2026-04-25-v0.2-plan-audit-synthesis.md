**Surface:** `docs/adrs/ADR-0005-multi-lens-v0.2-substrate.md:31`, `.planning/milestones/v0.2-MILESTONE.md:15`, `.planning/REQUIREMENTS.md` MCP-08 (around line 134).

**Proposed edit (canonical statement at ADR-0005:31):**
> `lens=` parameter on the four discovery MCP tools (`search_papers`, `browse_recent`, `find_related_papers`, `get_paper`). When `lens=` is omitted, behavior is single-lens semantic, preserving v0.1 response shape for backward compatibility. This default is a backward-compat decision flagged for sunset review after Phase 17 pilot signals are interpreted (LPILOT-03 milestone gate). "All available" considered as a UX-richer default and deferred — `v0.1` callers expect single-lens response shape.

**Cross-references:** MILESTONE.md:15 cites the ADR statement rather than restating it; REQUIREMENTS.md MCP-08 cites the ADR statement.

**Reasoning:** Convergent finding across all four audits. The current fork is the silent-defaults anti-pattern (`LONG-ARC.md:48`) operationalized at the document layer. One canonical statement, two cross-references.

### A2. Specify multi-value `lens=` behavior at architectural-commit layer (Finding #2)

**Surface:** `docs/adrs/ADR-0005-multi-lens-v0.2-substrate.md` (extend the `lens=` paragraph at :31).

**Current state:** No document specifies what happens for `lens=["semantic", "citation_community"]` without `mode=`. Phase 16 covers explicit lens-disagreement and intersection ops, but the multi-value form of an ordinary discovery query is unspecified.

**Proposed edit (extension to ADR-0005:31):**
> Multi-value `lens=` (a list of lens names) without an explicit `mode=` parameter returns per-lens result *dict* keyed by lens name: `{"semantic": [...results...], "citation_community": [...results...]}`. This is *not* fusion. Callers requesting fusion must pass `mode="fusion"` explicitly; callers requesting intersection or disagreement use Phase 16's dedicated operations. The default of per-lens dict honors ADR-0001's coexistence commitment at the API contract layer and the LONG-ARC anti-pattern against silent fusion-by-default (`LONG-ARC.md:46`, `:48`).

**Cross-reference:** Add Phase 13 success criterion in ROADMAP.md (around :282-285): "Multi-value `lens=` semantics implemented per ADR-0005's per-lens-dict default; explicit `mode=` parameters route to Phase 16 ops; fusion is opt-in via `mode='fusion'`."

**Reasoning:** Independent xhigh and high SV both flag this as blocking. Independent xhigh frames it as silent-default-by-absence: "When a Phase 12 or 13 implementer hits `search_papers(lens=["semantic", "citation_community"])` for the first time, they will pick a default behavior … that default will become the de facto API." The per-lens-dict default is the most multi-lens-honoring choice — it forces consumers to handle multi-lens results explicitly rather than collapsing them to a single ranked list.

### A3. Multi-attribution capture in Phase 17 (Finding #3 + #20)

**Surface:** `.planning/REQUIREMENTS.md:129` (LPILOT-02), `.planning/ROADMAP.md:353-354` (Phase 17 success criteria).

**Current state:** REQUIREMENTS.md:129: "Harness captures triage events with timestamps: selection, dismissal, return-to-paper after dismissal, lens-of-record per event." ROADMAP.md:353: "per-event lens-of-record."

**Proposed edits:**

REQUIREMENTS.md:129 (LPILOT-02):
> Harness captures triage events with timestamps: selection, dismissal, return-to-paper after dismissal. **For each event, captures the set of lens(es) that surfaced the paper for that event** (multi-valued; never assumes one-lens-per-event), plus the user's explicit lens selection if any (separate field). Multi-lens attribution preserved through analysis — the analysis layer must not reduce events to a single lens-of-record.

ROADMAP.md:353-354 (Phase 17 plan 1 success criteria):
> Capture schema designed: session-level lens usage, query log with lens(es) selected, triage events (selection, dismissal, return-to-paper), per-event multi-lens attribution (set-valued, not singular), durable storage, exportability. **Capture-schema change-control rule: mid-pilot schema additions are allowed; mid-pilot reinterpretation of existing fields is forbidden. Schema is versioned; the end-of-window artifact reports against schema version.**

**Reasoning:** Convergent finding (high SV + both xhigh runs). The "lens-of-record" formulation pre-commits the pilot's analysis to single-lens attribution, which is exactly the tournament framing 008 was superseded for (`MILESTONE.md:30`). The change-control rule (from contaminated xhigh's #20, surviving primary scrutiny) prevents the silent-redefinition failure mode mid-pilot — when the data starts arriving, there will be pressure to "reinterpret" existing fields to answer the questions the analyst wishes had been asked.

## 5. Methodology amendment

The audit comparison documented (in §3) measurable contamination effects when a same-vendor adversarial dispatch includes other audits as references. The methodology amendment proposed:

**Proposed addition to METHODOLOGY.md:112:**
> **Concrete change (extension):** Same-vendor adversarial dispatches must not include other audits of the same artifact set as references, even at lower effort levels or earlier dates. The dispatching prompt must include an explicit forbidden-reading list naming all prior audits of the same artifact set; the agent's compliance is verifiable in its tool-use trace. The contaminated 2026-04-25 xhigh dispatch (preserved at `.planning/audits/2026-04-25-v0.2-plan-audit-opus-adversarial-xhigh-contaminated.md`) is the documented evidence base.

**Status when committed:** Hypothesis (per CLAUDE.md status markers). Single-comparison evidence is suggestive, not strong. Mark as Hypothesis until a second methodology-audit cycle either confirms or weakens the pattern.

**Recommendation:** Adopt. The contamination cost is real and documented; the amendment is operationally cheap (a forbidden-reading list in the prompt); the cost of NOT amending is recurrence in the next paired-audit cycle.
