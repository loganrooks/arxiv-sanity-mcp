# Cross-Vendor Audit — v0.2 Plan + VISION + LONG-ARC + METHODOLOGY

## Summary

The integrated v0.2 artifact set is substantially coherent: ADR-0005's multi-lens commit is mapped into requirements, milestone success criteria, and Phases 12-17 with concrete implementation surfaces (`docs/adrs/ADR-0005-multi-lens-v0.2-substrate.md:28`, `.planning/REQUIREMENTS.md:105`, `.planning/ROADMAP.md:258`). I do not read a covert return to tournament narrowing or fusion-by-default in the main plan; the plan repeatedly names coexistence, per-lens explanations, disagreement, and intersection as first-class surfaces (`.planning/milestones/v0.2-MILESTONE.md:20`, `.planning/ROADMAP.md:332`). The main gaps are narrower: default-lens semantics differ between ADR-0005 and the plan, the Phase 14 OpenAlex coverage spike lacks a pre-registered decision threshold, and Phase 17 records capture-schema gaps without saying how mid-pilot schema changes are governed (`docs/adrs/ADR-0005-multi-lens-v0.2-substrate.md:31`, `.planning/ROADMAP.md:299`, `.planning/ROADMAP.md:356`).

## Findings by dimension

### Dimension 1 — LONG-ARC anti-pattern check

Finding: The plan mostly counters tournament narrowing and accidental single-lens interface. v0.2 explicitly ships two lenses, semantic plus citation/community (`.planning/milestones/v0.2-MILESTONE.md:10`), and Phase 15 requires a working `CitationCommunityLens` plus a third-lens design walkthrough before Phase 16 (`.planning/ROADMAP.md:317`, `.planning/ROADMAP.md:322`). This aligns with LONG-ARC's warning against validating an interface with only one lens (`.planning/LONG-ARC.md:47`). Confidence: high. Recommended action: keep Phase 15 plan 3 as a hard pre-Phase-16 gate, because it is the artifact that prevents the abstraction from being merely two-lens-local.

Finding: Default behavior is the clearest anti-pattern risk. ADR-0005 says `lens=` defaults to a configurable preferred lens or to "all available" (`docs/adrs/ADR-0005-multi-lens-v0.2-substrate.md:31`), while the milestone says default to semantic for backward compatibility (`.planning/milestones/v0.2-MILESTONE.md:15`) and Roadmap Phase 13 says omitted `lens=` matches v0.1 (`.planning/ROADMAP.md:282`). This is not an implementation violation yet, but it is a documented semantic fork in the authority surface. Confidence: high. Recommended action: before Phase 13 execution, record a chosen-for-now default policy: omitted `lens=` means v0.1 semantic-compatible behavior; explicit `lens=["semantic","citation"]` enters multi-lens operation; any "all available" default is deferred or exposed under an explicit mode.

Finding: Phase 16 does not silently default to fusion. It states the operations are "not fusion" and preserve per-lens results (`.planning/ROADMAP.md:332`), while LONG-ARC says fusion is not the default (`.planning/LONG-ARC.md:69`). Confidence: high. Recommended action: in 16-01, require tests or snapshots proving multi-lens list inputs do not collapse into RRF/fusion unless an explicit fusion mode is selected.

Finding: Phase 17 names opt-in explicitly, including "no silent profile mutation" (`.planning/ROADMAP.md:354`), which directly counters the silent-defaults anti-pattern (`.planning/LONG-ARC.md:48`). Confidence: high. Recommended action: define the opt-in mechanism in 17-01 with an auditable user action and storage flag, not only a prose promise.

### Dimension 2 — VISION coherence check

Finding: The v0.2 plan honors the VISION's product identity as a multi-lens substrate rather than a single ranked list. VISION says the center of gravity is multi-lens operations across orientation, lineage, dissent, longitudinal memory, provenance, and steering (`.planning/VISION.md:20`); the milestone ships two lenses, disagreement/intersection MCP operations, and a longitudinal pilot harness (`.planning/milestones/v0.2-MILESTONE.md:10`). Confidence: high. Recommended action: no major scope correction.

Finding: The plan does not foreclose named future operations. VISION identifies author/affiliation, benchmark/dataset, methodological, temporal, and behavior-derived lenses as open or speculative (`.planning/VISION.md:62`, `.planning/VISION.md:66`); the milestone explicitly leaves third lens, methodological taxonomy, behavior-derived signals, and broader pilots out of v0.2 scope (`.planning/milestones/v0.2-MILESTONE.md:32`). Confidence: high. Recommended action: preserve those as explicit non-actions in phase plans when nearby implementation choices could accidentally close them.

Finding: The plan does not commit to VISION's anti-vision targets. It rejects fusion-by-default (`.planning/milestones/v0.2-MILESTONE.md:20`), avoids paper-chat framing, and keeps discovery/triage operations central through MCP tools and pilot capture (`.planning/ROADMAP.md:278`, `.planning/ROADMAP.md:349`). Confidence: medium, because source-code/tool UX could still drift later. Recommended action: Phase 13 and 16 tool descriptions should be reviewed against VISION's "not a ranked-list-but-fancier" clause (`.planning/VISION.md:80`).

### Dimension 3 — LONG-ARC protected-seams check

Finding: All protected seams are represented, though not equally specified. Lens extensibility maps to Phase 12 registry work and Phase 15 third-lens walkthrough (`.planning/ROADMAP.md:264`, `.planning/ROADMAP.md:322`). Bundle-of-signals maps to LENS-04 and milestone commitment that new signal types are registry-added rather than migration-added (`.planning/REQUIREMENTS.md:110`, `.planning/milestones/v0.2-MILESTONE.md:17`). Per-lens provenance maps to `SearchResult`, `RankerSnapshot`, and citation-edge provenance (`.planning/ROADMAP.md:267`, `.planning/ROADMAP.md:284`, `.planning/ROADMAP.md:301`). Confidence: high. Recommended action: keep a seam checklist in each phase verification artifact.

Finding: Citation/community as first-class data is concretely instantiated in Phase 14 and Phase 15: retrieval-shaped edges, backfill, ingestion path, traversal, co-citation neighborhoods, and freshness handling (`.planning/ROADMAP.md:295`, `.planning/ROADMAP.md:318`, `.planning/ROADMAP.md:321`). The edges-table versus projection choice is deferred but bounded by the requirement "not write-once JSONB" (`.planning/REQUIREMENTS.md:115`). Confidence: high. Recommended action: require 14-02 to record the choice criteria before picking the schema shape.

Finding: Longitudinal state is present but thinner than the retrieval seams. Phase 17 covers session-level lens usage, query logs, triage events, lens-of-record, durable storage, and exportability (`.planning/ROADMAP.md:349`, `.planning/ROADMAP.md:353`), aligning with LONG-ARC's durable longitudinal-state seam (`.planning/LONG-ARC.md:38`). It does not yet specify how pilot capture interacts with existing workflow state or profile evolution. Confidence: medium. Recommended action: 17-01 should classify capture fields as settled/chosen-for-now/open and explicitly state which fields do not mutate profile state.

### Dimension 4 — METHODOLOGY practice-discipline check

Finding: The plan applies calibrated language better than the displaced tournament frame: OpenAlex coverage is named as empirical risk, not settled blocker (`.planning/milestones/v0.2-MILESTONE.md:79`), and pilot signal noise is accepted as density-over-isolation rather than overclaimed as causal value evidence (`.planning/milestones/v0.2-MILESTONE.md:90`). This matches METHODOLOGY's requirement that calibration occur where claims are made (`.planning/spikes/METHODOLOGY.md:139`, `.planning/spikes/METHODOLOGY.md:143`). Confidence: high. Recommended action: continue this register inside 14-01 and 17-02 artifacts.

Finding: The OpenAlex coverage spike is not pre-registered enough. Phase 14 requires edge density, freshness profile, and missingness characterization for 126 papers (`.planning/ROADMAP.md:299`), but no artifact defines the threshold for "coverage too thin" or what decision follows. METHODOLOGY warns that findings can be auxiliary-dependent and should vary evaluation/profile/sample assumptions where load-bearing (`.planning/spikes/METHODOLOGY.md:89`, `.planning/spikes/METHODOLOGY.md:93`). Confidence: high. Recommended action: before running 14-01, pre-register minimum viable coverage criteria, fallback branches, and whether the threshold governs storage shape, lens scope, or v0.2 release gating.

Finding: Paired-review and model-verification disciplines are acknowledged at milestone level (`.planning/milestones/v0.2-MILESTONE.md:95`) but not operationalized per phase. METHODOLOGY says audit-level remedies should not be adopted from single-reader contestable framing (`.planning/spikes/METHODOLOGY.md:112`) and gating audits should record reviewer model (`.planning/spikes/METHODOLOGY.md:127`). Confidence: medium. Recommended action: attach paired-review/model metadata requirements to Phase 15 walkthrough and Phase 17 pilot interpretation, where framing claims are likely.

### Dimension 5 — Critical-path realism check

Finding: The stated critical path is slightly compressed. The milestone lists 12 → 13 → 16 → 17, with 14 and 15 parallel where possible (`.planning/milestones/v0.2-MILESTONE.md:65`), but Phase 16 depends on both Phase 13 and Phase 15 (`.planning/ROADMAP.md:333`), and Phase 15 depends on Phase 12 and Phase 14 (`.planning/ROADMAP.md:314`). The real dependency chain is 12 plus 14 feeding 15, then 13 plus 15 feeding 16, then 17. Confidence: high. Recommended action: revise the critical-path line to `12 + 14 → 15; 12 → 13; 13 + 15 → 16 → 17`.

Finding: "~2 months" is plausible but contingent. The plan has 15 plans (`.planning/milestones/v0.2-MILESTONE.md:65`) and ADR-0005 says the target is about two months of engineering plus spike work (`docs/adrs/ADR-0005-multi-lens-v0.2-substrate.md:38`). The highest schedule risk is not plan count; it is uncertainty in Phase 14 coverage/schema and Phase 16 MCP operation shape (`.planning/ROADMAP.md:299`, `.planning/ROADMAP.md:336`). Confidence: medium. Recommended action: add a milestone checkpoint after 14-01 and 16-01 to decide whether v0.2 scope remains two-month-sized.

### Dimension 7 — Risk completeness

Finding: The six named risks are real and relevant: OpenAlex coverage, `SearchResult` ripple, two-lens validation limit, freshness drift, pilot signal noise, and tournament recurrence are all tied to plan surfaces (`.planning/milestones/v0.2-MILESTONE.md:86`, `.planning/milestones/v0.2-MILESTONE.md:91`). Confidence: high. Recommended action: keep them.

Finding: Missing or under-specified risks: backward-compat is named for `SearchResult` but should explicitly include existing MCP consumers, CLI flags, and tool docstrings because Phase 13 extends CLI and MCP surfaces (`.planning/milestones/v0.2-MILESTONE.md:87`, `.planning/ROADMAP.md:285`). Schema choice risk is named as a decision but not governed by criteria (`.planning/ROADMAP.md:300`). Phase 17 captures schema gaps at week 2 but lacks a change-control strategy (`.planning/ROADMAP.md:356`). Confidence: high. Recommended action: add these four as explicit risks or decision checkpoints.

Finding: Pilot signal isolation is already named and accepted in v0.2 (`.planning/milestones/v0.2-MILESTONE.md:90`). That is honest, but it means Phase 17 should not be used to claim lens value in isolation. Confidence: high. Recommended action: require the end-of-window artifact to separate "usage evidence," "friction evidence," and "value hypotheses."

## Cross-cutting observations

The plan is strongest where it converts doctrine into phase gates: Phase 12 for substrate, Phase 15 for second-lens validation, Phase 16 for disagreement/intersection, Phase 17 for opt-in longitudinal capture (`.planning/ROADMAP.md:258`, `.planning/ROADMAP.md:311`, `.planning/ROADMAP.md:330`, `.planning/ROADMAP.md:347`). The recurring weakness is decision governance inside spikes or design-choice plans: 14-01, 14-02, 16-01, and 17-01 all defer choices with rationale requirements, but only some define what evidence controls the choice (`.planning/ROADMAP.md:299`, `.planning/ROADMAP.md:300`, `.planning/ROADMAP.md:336`, `.planning/ROADMAP.md:353`).

The artifact set also shows one stale-doc issue: LONG-ARC says the v0.2 detailed plan "does not yet exist" (`.planning/LONG-ARC.md:26`), while the audited milestone and Roadmap now contain the v0.2 plan (`.planning/milestones/v0.2-MILESTONE.md:54`, `.planning/ROADMAP.md:31`). This is not a plan defect, but it is a planning-canon freshness issue. Recommended action: update LONG-ARC line 26 or mark it as pre-plan historical posture.

## Confidence calibration

High-confidence findings are those cross-confirmed across ADR-0005, requirements, milestone, and Roadmap: the mapping of ADR commitments, anti-fusion posture, two-lens validation structure, and explicit Phase 17 opt-in. The claims most warranting same-vendor adversarial confirmation are the default-lens semantic fork and the "not ranked-list-but-fancier" UX risk, because both depend on how wording becomes implementation and user-facing tool shape.
