<decisions_made>

- Phase 10 was completed on main branch before branching for spike work
- The deployment deliberation is intentionally left OPEN — not concluded until spike findings provide sufficient evidence
- The "tier" model (Bronze/Silver/Gold) was proposed but is a hypothesis, not a decision — the user pushed back on premature conclusion
- Deployment mode (local vs hosted) is orthogonal to capability tier — hosting any tier is possible
- The research program follows inquiry methodology, not verificationism — we gather data to understand what's happening, not to confirm claims
- The NLP layer (TF-IDF, SVM, embeddings) runs in Python, not in the database — this was a key insight that narrowed the PostgreSQL advantage to: indexed ANN at >500K vectors, concurrent multi-writer, and mature extensions
- The user wants transparency for informed user decisions at install time, not just automatic hardware detection

The user (Logan, philosophy PhD) is thinking deeply about epistemological rigor in experimental design, product architecture for varying hardware constraints, and how to offer both simplicity and power. They want the system to be genuinely useful for philosophy-of-AI research, not just technically functional.

Key unresolved design questions from the conversation:
- Feature lifecycle management (where do pre-computed TF-IDF/embedding matrices live between MCP server restarts?)
- Capability system vs fixed tiers (the user wants both tiers AND custom configurations)
- "Project" as a first-class concept (different recommenders for different research projects)
- Cold start problem (recommendations before enough triage data exists)
- Migration paths between configurations (not just rows — also feature matrices, indexes)
- Multi-machine continuity (dionysus workstation vs apollo MacBook)
