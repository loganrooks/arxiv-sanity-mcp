## Spike program purpose reframed
Spikes characterize the trade-off space for different user situations. Output is a map of options with honest trade-off descriptions, not a verdict about which model wins. Different users (local-first vs quality-maximizing, GPU vs CPU-only, CS/ML vs interdisciplinary) will choose different configurations from the same characterization.

1. Spikes are not competitions — they characterize options for users with different situations
2. Representation is not function — knowing models produce different rankings says nothing about which rankings are better for researchers
3. Every evaluation method constitutes the phenomenon it measures — Jaccard, tau, AI review, agent task each reveal different aspects

<anti_pattern>
THE SPECIFIC LESSON FROM THIS SESSION: When a design identifies a problem (MiniLM-centrism, Jaccard gatekeeping, evaluator monoculture), the execution must ACTUALLY ADDRESS IT — not just acknowledge it in a caveat section. Spike 004's DESIGN.md correctly identified MiniLM-entanglement. PROTOCOL.md added mitigations. The execution still treated MiniLM as ground truth. The synthesis still led with J@20.

Self-awareness without structural enforcement produces Pattern A. The independent design critic (principle 4) exists to catch this BEFORE execution, not after. If the critic identifies a problem and the design revision only acknowledges it rather than structurally addressing it, the critic should escalate.

The codebase does NOT implement dense embeddings. All embedding work is in spike experiment scripts. Agent-based evaluation (H1, H5) requires building task infrastructure, not just running experiments on existing data.
</anti_pattern>
