# Milestone v0.2: Multi-Lens MCP Substrate

**Status:** Planned (kicked off 2026-04-25)
**Target completion:** ~2 months from execution start
**Architectural commit:** [ADR-0005](../../docs/adrs/ADR-0005-multi-lens-v0.2-substrate.md)
**Implementation phases:** 12-17

## Goal

Convert the multi-lens redirection committed in ADR-0005 into a working substrate. v0.2 ships at least two lenses (existing semantic + new citation/community), generalizes the profile primitive into a bundle-of-signals shape, exposes lens-disagreement and intersection as first-class MCP operations, and replaces the superseded `008` tournament with a longitudinal pilot harness running with one user.

## Architectural commitments (from ADR-0005)

- **`Lens` interface** — registered implementations producing `query(seed_or_profile, options) → ranked_results_with_provenance`. Module-level registry; plugin-style registration.
- **`lens=` parameter** on the four discovery MCP tools (`search_papers`, `browse_recent`, `find_related_papers`, `get_paper`). Default and multi-value semantics per ADR-0005 (single-lens semantic when omitted; per-lens dict when a list is passed without explicit `mode=`).
- **`ProfileRankingService` becomes a lens dispatcher** rather than a single hardcoded pipeline.
- **Bundle-of-signals profile primitive** — `signal_type` already open at the DB level (post-migration 005); new signal types added by registry, not by schema migration.
- **Citations storage** — retrieval-shaped (normalized edges or denormalized projection from `PaperEnrichment.related_works`). Provenance fields per ADR-0003.
- **`SearchResult` shape generalized** — carries per-lens score components and per-lens explanations alongside legacy composite. Backward-compatible: existing consumers continue to receive composite `score` and `ranking_explanation`.
- **Lens-disagreement and intersection** as MCP-exposed operations. Steering, intersection, lens-disagreement, per-paper cross-lens explanation are first-class. Fusion is one strategy among several, not the implicit default.

## Scope: in vs out

**In scope (v0.2):**
- Lens abstraction primitives
- Citation/community lens (OpenAlex-backed; Semantic Scholar adapter deferred)
- MCP surface lens-awareness
- Lens-disagreement and intersection ops
- Longitudinal pilot harness with one user (Logan)
- 007 override annotation and 008 supersession (already complete)

**Out of scope (deferred to v0.3 or later):**
- Third lens (author/affiliation, benchmark/dataset, methodological)
- Semantic Scholar adapter (second citation source)
- Behavior-derived profile signals
- Multi-user / collaborative scenarios
- Methodological-lens taxonomy work
- pgvector or graph-DB backend introduction
- Full-text content for citation-lens explanations
- Broader pilot beyond Logan

## Milestone-level success criteria

1. Two lenses (semantic + citation/community) ship with structurally distinct retrieval logic and per-lens explanations.
2. Adding a hypothetical third lens (e.g., author/affiliation) does not require modifying any existing consumer code, validated by design walkthrough during Phase 15.
3. The four discovery MCP tools accept a `lens=` parameter; backward-compat tests confirm default behavior is unchanged for existing consumers.
4. Lens-disagreement and intersection are exposed via MCP and work correctly across the two shipped lenses.
5. Longitudinal pilot harness captures lens usage, selections, dismissals, and returns; pilot runs continuously through Logan's research practice for at least four weeks during or after Phase 17 plans complete.
6. ADR-0001's coexistence commitment is honored at the implementation layer, not only at the design layer.
7. All new ranking/retrieval signals carry provenance per ADR-0003.

## Open scoping calls (resolved as of 2026-04-25)

- **Citation source: OpenAlex first.** Semantic Scholar deferred to v0.3 as a second adapter. Empirical risk: OpenAlex's coverage of arXiv AI papers may be too thin for v0.2 lens-validation; Phase 14's first plan includes a coverage spike to characterize this on the 126-paper corpus before committing to storage shape.
- **Pilot scope: Logan only.** v0.3 broadens to 2-3 external AI-research pilots if v0.2 closes its loops cleanly.
- **`SearchResult` backward compat: hide new shape behind default.** Legacy `score` and `ranking_explanation` keep current meaning (composite over the active lens); `lens_scores` and `per_lens_explanations` are additive opt-in fields.
- **Phase 6 (content normalization) pull-forward: not needed.** Content normalization is already complete in v0.1; the citation lens does not depend on full-text content.

## Risks

- **OpenAlex coverage risk** (named above). Mitigation: coverage spike in Phase 14 plan 1; pre-registered decision matrix in Phase 14 plan 1 specifies fallback branches (proceed / flag-for-v0.3 / re-deliberate) before the spike runs, preventing post-hoc sunk-cost interpretation of "this is sufficient" (per audit synthesis 2026-04-26).
- **`SearchResult` shape change ripple.** Generalization touches all retrieval consumers: CLI flag interfaces, CLI output formatters, MCP tool signatures, MCP tool docstrings, MCP resource schemas, and the v0.1 test suite (403 tests, of which an audit subset specifically covers retrieval response shape). Mitigation: backward-compat default per ADR-0005; Phase 12 plan 2 includes regression tests preserving v0.1 response shape; Phase 13 plan 1 includes an audit pass on existing CLI flag and MCP tool docstring surfaces, with an explicit list of any user-facing wording changes recorded as part of the phase artifact.
- **Lens-extensibility validated only by two lenses, not three.** ADR-0005 acknowledges this — the abstraction is most likely right when validated by a second implementation, but a third would harden it further. Mitigation: Phase 15 plan 3 is a design walkthrough for a hypothetical third lens, surfacing abstraction gaps without implementing.
- **Citation graph freshness drift.** OpenAlex updates continuously; backfilled edges go stale. Mitigation: provenance fields per ADR-0003 record retrieval timestamps; Phase 14 schema includes a freshness window.
- **Longitudinal pilot signal noise.** Multi-lens pilot may capture lens *usage* but not isolate lens *value*. Acceptable in v0.2 (signal density over isolation); v0.3 may add controlled probes alongside pilot if isolation becomes urgent.
- **Tournament-narrowing pattern recurrence.** The redirection is committed but the pattern is recurrent; Phase 12-15 work could quietly drift toward "which lens wins" if not actively counter-postured. Mitigation: LONG-ARC.md anti-pattern watch *operationalized* at Phase 15 plan 3 (lens-extensibility design walkthrough) and Phase 16 plan 1 (lens-disagreement API design) — see ROADMAP success criteria for both. Vigilance is what failed in 005-008; mitigating recurrence with vigilance alone recapitulates the failure mode, so the counter-posture is structural (phase-boundary checkpoints).
