## User Decisions

### 1. Pre-MCP Quality Fixes (do first, before MCP wiring)

Two known defects must be fixed before exposing services via MCP, since the first real MCP usage shouldn't be undermined by known issues:

**PREMCP-01: Fix ranking triple-counting (CRITICAL)**
- Category Jaccard is computed 3 times: `score_category_overlap` (weight 0.15) + inside `score_seed_relation` (60% of 0.25 = 0.15) + inside `score_profile_match` (50% of 0.15 = 0.075)
- Effective category weight: **0.375** (intended: 0.15)
- Fix: Remove redundant Jaccard from `score_seed_relation` and `score_profile_match`. Score category overlap exactly once via the dedicated scorer. Redistribute freed weight to seed-specific and profile-specific signals.
- File: `src/arxiv_mcp/interest/ranking.py`
- Tests: Verify effective category weight matches DEFAULT_WEIGHTS[CATEGORY_OVERLAP]

**PREMCP-02: Restore "seen" triage state**
- WKFL-03 intended to include "seen" but it was dropped. Absence-means-unseen is fine for `unseen`, but there's no way to mark "encountered, decision pending."
- Fix: Add `seen` to TriageState CHECK constraint. Add `mark_seen` service operation. Migration to update allowed values.
- File: `src/arxiv_mcp/db/models.py`, `src/arxiv_mcp/workflow/triage.py`

**PREMCP-03: Document over-fetch pagination limitation**
- `_ranked_search` uses 3x over-fetch + re-rank + truncate. After re-ranking, cursor pagination is approximate — page boundaries shift.
- Fix: Add docstring/comment documenting this as a known limitation. Not a code fix.

### 2. MCP Server Architecture

Per ADR-0004 (MCP as workflow substrate, not thin wrapper):

**Stack:** Use the MCP Python SDK (`mcp` package). The server wraps existing service classes (SearchService, CollectionService, TriageService, ProfileService, EnrichmentService) — no new business logic, just MCP protocol surface.

**Transport:** stdio (standard for Claude Code MCP servers). Server runs as a subprocess.

### 3. MCP Surface Design

Based on ecosystem analysis and literature review workflow needs:

**Tools (actions that change state):**
| Tool | Service | Notes |
|------|---------|-------|
| `search_papers` | SearchService | Lexical search with optional profile-based re-ranking |
| `browse_recent` | SearchService | Recently announced papers by category |
| `find_related_papers` | SearchService | From seed paper(s) via lexical similarity |
| `get_paper` | SearchService | Single paper lookup by arXiv ID |
| `triage_paper` | TriageService | Set triage state (shortlisted/dismissed/read/cite-later/archived/seen) |
| `add_to_collection` | CollectionService | Add paper to named collection (create if needed) |
| `create_watch` | SavedQueryService | Monitor a saved query for new results |
| `add_signal` | ProfileService | Add signal to interest profile |
| `enrich_paper` | EnrichmentService | Trigger OpenAlex enrichment |

Count: 9 tools (within MCP-07's 5-10 range)

**Resources (read-only state):**
| Resource URI | Service | Notes |
|-------------|---------|-------|
| `paper://{arxiv_id}` | SearchService + TriageService + EnrichmentService | Paper metadata + triage state + enrichment data |
| `collection://{slug}` | CollectionService | Collection contents with triage states |
| `profile://{slug}` | ProfileService | Interest profile with signals and weights |
| `watch://{slug}/deltas` | SavedQueryService | Papers new since last checkpoint |

**Prompts (deferred to Phase 5 validation):**
Prompt design requires real usage to validate. Phase 04.1 ships tools + resources only. Phase 5 adds prompts based on what workflows emerge.

### 4. Literature Review Feature Priorities (informational — for planner context)

Ordered by value for the first real user (arxiv-scan pipeline continuation):

1. **Multi-seed expansion with provenance** (Highest) — `find_related_papers` should support multiple seeds with tracking
2. **Cross-source relatedness surfacing** (High) — OpenAlex `related_works` already stored, not exposed as discovery tool
3. **Watch-based monitoring with enriched deltas** (High) — Auto-shortlist high-confidence papers from watch deltas
4. **Interest profile as scope document** (Medium) — Add description field, profile snapshots
5. **BibTeX/citation export** (Medium) — Low cost, arXiv metadata has the fields
6. **Bulk triage operations** (Lower) — Import existing triage decisions, mark multiple papers

Items 1-3 should be considered during Phase 04.1 planning if they don't add excessive scope.
Items 4-6 are Phase 5 (validation iteration) candidates.

### 5. Cross-Project Data Flow (informational — future phase)

After MCP v1 ships, there's a data import path from the arxiv-scan pipeline (/scratch/arxiv-scan/):
- 150 analyzed papers → MCP paper metadata + triage states
- 10 tension definitions → interest profile signals
- 1,211 triage decisions → bulk triage import (ranking validation)

This import is Phase 5 work, not Phase 04.1. But the Phase 04.1 MCP surface must support the operations this import will need (search, triage, collection, profile signals).

### 6. What NOT to build in Phase 04.1

- No content tools (MCP-03 deferred to Phase 6)
- No prompts (MCP-05 deferred to Phase 5)
- No semantic search (v2)
- No new business logic — MCP wraps existing services only
