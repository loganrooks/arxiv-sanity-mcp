## Summary

Phase 04.1 has two distinct workstreams: (1) three pre-MCP quality fixes to the existing codebase, and (2) building an MCP server that exposes existing services as tools and resources. The pre-MCP fixes are surgical and well-scoped: the ranking triple-counting bug is clearly visible in the `score_seed_relation` and `score_profile_match` functions, the "seen" triage state requires a DB migration and minor service update, and the pagination documentation is a docstring addition.

The MCP server workstream uses the official MCP Python SDK (`mcp` package, v1.26.0) which includes FastMCP -- the high-level server framework. FastMCP provides decorator-based tool and resource registration (`@mcp.tool()`, `@mcp.resource()`), automatic Pydantic schema generation from type annotations, and a lifespan pattern for managing database connections. The server wraps existing service classes with no new business logic. All 9 tools and 4 resources map directly to existing service methods.

**Primary recommendation:** Use `mcp>=1.25,<2` (pin to v1.x) with FastMCP's `@mcp.tool()` and `@mcp.resource()` decorators, a lifespan context manager for SQLAlchemy engine/session factory initialization, and stdio transport. Serialize all Pydantic model returns to dicts via `.model_dump()` before returning from tools to avoid serialization edge cases.

### Pattern 1: Lifespan Context for Database Connection

**What:** Use FastMCP's lifespan pattern to create and tear down the SQLAlchemy async engine and session factory.
**When to use:** Always -- the MCP server needs a DB connection for all operations.

```python
from contextlib import asynccontextmanager
from collections.abc import AsyncIterator
from dataclasses import dataclass

from sqlalchemy.ext.asyncio import AsyncEngine, async_sessionmaker, AsyncSession
from mcp.server.fastmcp import FastMCP

from arxiv_mcp.config import get_settings
from arxiv_mcp.db.engine import create_engine, session_factory


@dataclass
class AppContext:
    """Typed lifespan context holding initialized services."""
    engine: AsyncEngine
    session_factory: async_sessionmaker[AsyncSession]
    # Service instances go here


@asynccontextmanager
async def app_lifespan(server: FastMCP) -> AsyncIterator[AppContext]:
    """Initialize DB engine and services on startup, dispose on shutdown."""
    settings = get_settings()
    engine = create_engine(settings.database_url)
    sf = session_factory(engine)
    try:
        yield AppContext(engine=engine, session_factory=sf)
    finally:
        await engine.dispose()


mcp = FastMCP("arxiv-mcp", lifespan=app_lifespan)
```

### Pattern 2: Service-Delegating Tools

**What:** Each tool function gets the lifespan context, instantiates or retrieves the appropriate service, calls its method, and serializes the result.
**When to use:** All tool implementations.

```python
from mcp.server.fastmcp import Context

@mcp.tool()
async def search_papers(
    query: str,
    category: str | None = None,
    days: int | None = None,
    page_size: int = 20,
    cursor: str | None = None,
    ctx: Context = None,
) -> dict:
    """Search for papers by text query, with optional category and date filters."""
    app = ctx.request_context.lifespan_context
    svc = SearchService(app.session_factory, get_settings())
    result = await svc.search_papers(
        query_text=query,
        category=category,
        page_size=page_size,
        cursor_token=cursor,
    )
    return result.model_dump(mode="json")
```

### Pattern 4: Pydantic Model Serialization for Tool Returns

**What:** All tools return `dict` (not Pydantic models) via `.model_dump(mode="json")`. This ensures clean JSON serialization without Pydantic v2 edge cases.
**When to use:** All tool return values that involve Pydantic models.

```python
@mcp.tool()
async def triage_paper(arxiv_id: str, state: str, reason: str | None = None, ctx: Context = None) -> dict:
    """Set the triage state for a paper (seen, shortlisted, dismissed, read, cite-later, archived)."""
    app = ctx.request_context.lifespan_context
    svc = TriageService(app.session_factory, get_settings())
    result = await svc.mark_triage(arxiv_id, state, source="mcp", reason=reason)
    return result.model_dump(mode="json")
```

### Anti-Patterns to Avoid
- **Business logic in tools:** Tools MUST only delegate to existing services. No query building, no SQL, no new algorithms. If the service doesn't support an operation, it belongs in a future phase.
- **Returning raw Pydantic models from tools:** Use `.model_dump(mode="json")` to avoid MCP SDK serialization issues with Pydantic v2.
- **Creating services per-request without caching:** While the current services are stateless (they just hold session_factory and settings), avoid repeated heavy initialization. Consider caching service instances in AppContext.
- **Mixing concerns in a single file:** Keep tool definitions separate from resource definitions. Each maps to a service domain.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| MCP protocol handling | Custom JSON-RPC handler | `mcp` SDK (FastMCP) | Protocol compliance, schema generation, transport handling |
| Tool schema generation | Manual JSON Schema | FastMCP auto-generation from type hints | Pydantic-based, automatic, correct |
| Resource URI routing | Custom URI parser | `@mcp.resource("uri://{param}")` pattern | FastMCP handles template matching |
| Server lifecycle | Manual signal handling | FastMCP `lifespan` + `mcp.run()` | Clean startup/shutdown with typed context |
| Slug generation | New slug function | Existing `workflow.util.slugify` | Already tested and used throughout |

**Key insight:** The entire MCP layer is a thin translation from MCP protocol to existing service method calls. Zero business logic in the MCP module.

## Common Pitfalls

### Pitfall 1: Ranking Triple-Counting Fix Breaks Existing Tests
**What goes wrong:** Removing category Jaccard from `score_seed_relation` and `score_profile_match` changes expected scores in ranking tests.
**Why it happens:** Tests assert specific composite scores that include the triple-counted category weight.
**How to avoid:** Update test expectations AFTER fixing scorers. Verify new effective category weight = DEFAULT_WEIGHTS[CATEGORY_OVERLAP] = 0.15 exactly.
**Warning signs:** Test failures in `tests/test_interest/test_ranking.py`.

### Pitfall 2: "Seen" State vs Absence-Means-Unseen Pattern
**What goes wrong:** The "seen" state is stored as a row (like "shortlisted"), but "unseen" is absence of a row. These are semantically different -- "seen" means "encountered, no decision yet" while "unseen" means "never encountered."
**Why it happens:** The absence-means-unseen pattern conflates "never seen" with "explicitly marked unseen."
**How to avoid:** "seen" creates a TriageState row (like other non-unseen states). Only "unseen" uses the absence-means-no-row pattern. The TriageService.mark_triage method already handles this correctly for all non-"unseen" states.
**Warning signs:** If `mark_triage("seen")` deletes the row instead of creating one.

### Pitfall 3: Lifespan Context Access Pattern
**What goes wrong:** Tools cannot access the database because lifespan context is accessed incorrectly.
**Why it happens:** FastMCP's context access pattern (`ctx.request_context.lifespan_context`) must match the lifespan yield type. If AppContext is a dataclass, access is via attribute; if dict, via key.
**How to avoid:** Use a typed dataclass for AppContext. Access via `ctx.request_context.lifespan_context.session_factory` (attribute access, not dict).
**Warning signs:** `AttributeError` or `KeyError` in tool functions.

### Pitfall 4: Alembic Migration Numbering
**What goes wrong:** Migration revision conflicts if multiple migrations share the same number.
**Why it happens:** Existing migrations go 001-006. The "seen" state migration must be 007.
**How to avoid:** Check existing migrations before creating. Use sequential numbering. Set `down_revision` correctly.
**Warning signs:** Alembic "multiple heads" error.

### Pitfall 5: Async Event Loop Conflicts in Tests
**What goes wrong:** MCP server tests conflict with existing test infrastructure event loops.
**Why it happens:** The MCP SDK runs its own event loop for stdio transport. Test fixtures use separate async engines.
**How to avoid:** Test MCP tools by calling their underlying functions directly (unit tests), not by starting a full MCP server. Use the same `test_engine` / `test_session` fixtures. For integration tests, use the `mcp` SDK's test client if available.
**Warning signs:** `RuntimeError: This event loop is already running`.

### Pitfall 6: find_related_papers Signature Mismatch
**What goes wrong:** The MCP tool `find_related_papers` needs to support multiple seed IDs (per CONTEXT.md feature priority 1), but `SearchService.find_related_papers` only accepts a single `seed_arxiv_id`.
**Why it happens:** The existing service was built for single-seed lookup.
**How to avoid:** For Phase 04.1, expose single-seed `find_related_papers` (matching existing service). Multi-seed expansion is a feature priority, not a requirement. If implementing, loop over seeds and merge/deduplicate results at the MCP layer (no service modification needed).
**Warning signs:** Adding new service methods when the phase explicitly says "no new business logic."

### Pitfall 7: Tool Parameter Names vs Service Parameter Names
**What goes wrong:** MCP tool parameter names don't match service method parameter names, causing mapping errors.
**Why it happens:** Tools use user-intent naming (e.g., `query`) while services use implementation naming (e.g., `query_text`).
**How to avoid:** Map tool params to service params explicitly in each tool function. Document the mapping.
**Warning signs:** `TypeError: unexpected keyword argument`.

### Example 1: PREMCP-01 Fix -- Remove Redundant Jaccard from score_seed_relation

```python
# BEFORE (ranking.py, score_seed_relation):
# Sub-signal 1: category overlap with seeds (REDUNDANT -- duplicates score_category_overlap)
paper_cats = set(paper.category_list or [])
cat_score = 0.0
if seed_categories and paper_cats:
    intersection = paper_cats & seed_categories
    union = paper_cats | seed_categories
    cat_score = len(intersection) / len(union) if union else 0.0
# Weighted composite (60% category, 40% author)
composite = 0.6 * cat_score + 0.4 * author_score

# AFTER: Remove category Jaccard, use only author match for seed_relation
# The category overlap signal is already handled by score_category_overlap
# seed_relation now focuses on seed-specific signals: author match
# (Future: could add title/abstract similarity sub-signal)
author_score = 0.0
if followed_authors:
    paper_authors = parse_authors(paper.authors_text or "")
    matched = sum(1 for a in paper_authors if a in followed_authors)
    author_score = min(1.0, matched / max(1, len(followed_authors)))
composite = author_score  # Pure author-match signal
```

### Example 4: MCP Server Entry Point

```python
"""MCP server for arxiv-mcp research discovery.

Exposes search, workflow, interest, and enrichment services as MCP
tools and resources via stdio transport.
"""

from contextlib import asynccontextmanager
from collections.abc import AsyncIterator
from dataclasses import dataclass

from sqlalchemy.ext.asyncio import AsyncEngine, async_sessionmaker, AsyncSession
from mcp.server.fastmcp import FastMCP

from arxiv_mcp.config import Settings, get_settings
from arxiv_mcp.db.engine import create_engine, session_factory
from arxiv_mcp.search.service import SearchService
from arxiv_mcp.workflow.collections import CollectionService
from arxiv_mcp.workflow.triage import TriageService
from arxiv_mcp.workflow.queries import SavedQueryService
from arxiv_mcp.workflow.watches import WatchService
from arxiv_mcp.interest.profiles import ProfileService
from arxiv_mcp.enrichment.service import EnrichmentService


@dataclass
class AppContext:
    """Typed context holding initialized services for tool/resource use."""
    engine: AsyncEngine
    session_factory: async_sessionmaker[AsyncSession]
    settings: Settings
    search: SearchService
    collections: CollectionService
    triage: TriageService
    saved_queries: SavedQueryService
    watches: WatchService
    profiles: ProfileService
    enrichment: EnrichmentService


@asynccontextmanager
async def app_lifespan(server: FastMCP) -> AsyncIterator[AppContext]:
    settings = get_settings()
    engine = create_engine(settings.database_url)
    sf = session_factory(engine)

    search_svc = SearchService(sf, settings)
    ctx = AppContext(
        engine=engine,
        session_factory=sf,
        settings=settings,
        search=search_svc,
        collections=CollectionService(sf, settings),
        triage=TriageService(sf, settings),
        saved_queries=SavedQueryService(sf, settings, search_svc),
        watches=WatchService(sf, settings, search_svc),
        profiles=ProfileService(sf, settings),
        enrichment=EnrichmentService(sf, settings),
    )
    try:
        yield ctx
    finally:
        await engine.dispose()


mcp = FastMCP("arxiv-mcp", lifespan=app_lifespan)

# Tool and resource registrations imported from submodules
from arxiv_mcp.mcp.tools import discovery, workflow, interest, enrichment  # noqa
```

### Example 6: Resource Template Implementation

```python
@mcp.resource("paper://{arxiv_id}")
async def paper_resource(arxiv_id: str, ctx: Context) -> dict:
    """Paper metadata including triage state and enrichment data.

    Combines data from search, triage, and enrichment services
    into a single canonical paper object.
    """
    app = ctx.request_context.lifespan_context
    settings = app.settings

    # Get paper from DB
    from sqlalchemy import select
    from arxiv_mcp.db.models import Paper
    async with app.session_factory() as session:
        result = await session.execute(
            select(Paper).where(Paper.arxiv_id == arxiv_id)
        )
        paper = result.scalar_one_or_none()
        if paper is None:
            return {"error": f"Paper not found: {arxiv_id}"}

    # Get triage state
    triage_state = await app.triage.get_triage_state(arxiv_id)

    # Get enrichment
    enrichment = await app.enrichment.get_enrichment_status(arxiv_id)

    # Get collections
    collections = await app.collections.get_paper_collections(arxiv_id)

    return {
        "arxiv_id": paper.arxiv_id,
        "title": paper.title,
        "authors": paper.authors_text,
        "abstract": paper.abstract,
        "categories": paper.category_list,
        "primary_category": paper.primary_category,
        "submitted_date": str(paper.submitted_date),
        "triage_state": triage_state,
        "collections": [c.model_dump(mode="json") for c in collections],
        "enrichment": enrichment.__dict__ if enrichment else None,
    }
```

## Open Questions

1. **Context access pattern for lifespan dataclass**
   - What we know: FastMCP lifespan yields a context object accessible via `ctx.request_context.lifespan_context`
   - What's unclear: Whether the lifespan context is the AppContext dataclass directly or wrapped in a dict. The official docs show both patterns.
   - Recommendation: Test with a minimal server first. If dataclass attributes aren't accessible, switch to dict access.

2. **`get_paper` tool -- single paper lookup**
   - What we know: CONTEXT.md lists `get_paper` as a tool, but there's no `get_paper` method on SearchService. The ORM supports `session.get(Paper, arxiv_id)`.
   - What's unclear: Should this be a tool (action) or just the `paper://{arxiv_id}` resource?
   - Recommendation: Implement both. The tool returns a focused summary; the resource provides the full canonical view with triage/enrichment/collection data. This follows MCP best practice (tools for actions, resources for data).

3. **Service caching in AppContext**
   - What we know: Services are stateless (hold session_factory + settings). Creating them per-request is cheap.
   - What's unclear: Whether FastMCP lifespan context is shared across requests or re-created.
   - Recommendation: Create services once in the lifespan (as shown in Example 4). They're cheap to hold and this matches the existing CLI pattern.
