## Decisions Made
- **Database user creation:** Created `arxiv_mcp` PostgreSQL user with password auth (not peer auth) since the application connects via asyncpg with explicit credentials.
- **Function-scoped engine fixtures:** Used function-scoped (not session-scoped) async engines in tests to avoid asyncpg event loop conflicts between test functions.
- **Split trigger SQL:** asyncpg does not support multiple statements in a single `execute()`, so tsvector trigger SQL is split into three separate statements (function, drop trigger, create trigger).
- **Hand-written migration:** Migration 001 is hand-written per research pitfall 5 about Alembic autogenerate instability with GIN indexes.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] asyncpg multi-statement SQL incompatibility**
- **Found during:** Task 2 (tsvector trigger test setup)
- **Issue:** asyncpg does not allow multiple SQL statements in a single prepared statement. The combined trigger SQL (CREATE FUNCTION + DROP TRIGGER + CREATE TRIGGER) failed.
- **Fix:** Split the single SQL block into three separate `execute()` calls in conftest.py.
- **Files modified:** tests/conftest.py
- **Verification:** All 9 paper model tests pass
- **Committed in:** c3b2b00 (Task 2 commit)
